数据结构

内存模型如下：

在 map 的内存模型中一共有三种结构: hmap , bmap , mapextra

  • hmap 表示整个 map，bmap 表示 hmap 中的一个桶，一个 hmap 有很多个桶
  • 当一个桶满了之后，指向的下一个桶叫做溢出桶，—> 拉链法
  • mapextra 表示所有的溢出桶，目的是为了用于 gc，避免 gc 时扫描整个 map，仅扫描所有的溢出桶就够了？
    • 如果 map 的 key和 value 都不包含指针的话，在 GC 期间就可以避免对它的扫描，在 map 非常大的情况下能提升性能
    • 如何实现不扫描的？
      • 在 bmap 中有一个 overflow 指针，指向溢出的 bucket，因为是指针所以 GC 时需要扫描，也就要扫描所有的 overflow 指针
      • 当 map 的key/value 是非指针类型时，扫描可以避免，直接标记整个 map 的颜色（三色标记）不需要扫描每个 bmap 的 overflow 指针
  • bmap 中有部分字段是在编译期间生成的，如 keys,values…
  • 在 bmap 中将所有的key 放在一起，value放在一起即 key/key/key/key/key…/value/value/value，目的是在某些情况下省去 pad 字段，节省内存空间
  • golang 中使用的内存是不会收缩的，只会越来越多

设计原理

hash 值的使用

通过 hash 函数，key 可以得到一个唯一值，map 将这个值分为高 8 位和低 8 位

  • 低 8 位：用于寻找当前 key 属于哪个 bucket
  • 高 8 位：用于寻找当前 key 在 bucket 中的位置，存储在 tophash 字段中

桶的结构的设计

bmap 即 桶

  • 桶是 map 中最小的挂载粒度：每 8 个 kv 键值对放在一个桶中，通过链表的形式串联起来，用于减少对象数量，减轻 gc 的负担
  • 桶串联实现拉链法：当某个桶满了，会申请一个新桶，挂在这个桶后面形成链表，新桶优先使用预分配的桶
  • 哈希高 8 位优化桶查找 key：查找时不用比较完整的 key 就能过滤掉不符合要求的 key，tophash 中的值相等，再去比较 key 值
  • 桶中 key/value 分开放：桶中所有的 key/value 放在一起，目的是方便内存对齐
  • 根据 k/v 大小存储不同值：当 k/v 大小超过 128 字节时，存储指针；当小于等于 128 字节时，存储的字段为原值
  • 桶的搬迁状态：tophash 字段的值是否小于 minTopHash 用于表示桶的搬迁状态

map 的扩容于搬迁策略

  1. 扩容策略
    • map 的扩容策略是新分配一个更大的数组，在插入和删除 key 的时候，将对应桶中的数据迁移到新分配的桶中去
  2. 搬迁策略
    • map 的扩容需要将原有的 kv 键值对迁移搬迁到新的内存地址，直接一下子全部搬完回影响性能
    • 采用渐近式搬迁策略，将搬迁的 O(N) 开销均摊到 O(1) 的赋值和删除操作上
  3. 扩容场景
    • 当装载因子超过6.5 时，扩容一倍
    • 当使用的溢出桶过多时，重新分配一样大的内存空间，属于等量扩容，实际上没有扩容，为了回收空闲的溢出桶
💡

装载因子等于 = map 中元素的数量 / len(map) 即 2^B

  • 装载因子越大表示空闲位置越小，冲突也越多
  • 随着装载因子的增大，当达到 100% 时，查找和插入任意元素的复杂度都是 O(N)
  1. 什么情况下会触发等量扩容
    • 溢出桶的数量大于等于
    • 1 >> B & min(15,B)
